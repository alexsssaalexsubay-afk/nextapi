# 批量生成指南

**批量任务**就是把一批视频生成任务写进 CSV 文件。一行 = 一条镜头。Batch Studio 负责提交任务、追踪进度、下载成品，并生成完整的运行记录。你只需要一次点击。

---

## 镜头清单 CSV 是什么

清单是批量任务的核心。把它理解成一张拍摄表：你在 Excel 或 Google Sheets 里填好，保存成 `.csv`，上传到 Batch Studio 就能跑。

### 一个最简可用的清单

```csv
shot_id,prompt_en,duration,aspect_ratio
ep01_s01_001,"林悦走进咖啡馆，在靠窗的位置停下来，望向窗外",5,16:9
ep01_s01_002,"林悦搅动咖啡的特写，刘海垂落",4,16:9
ep01_s01_003,"陈默走进咖啡馆，隔着人群看见林悦",5,16:9
```

三行，三条镜头。就这么简单。

### 必填列

每份清单**必须**包含这四列。缺少任何一列都会触发验证错误。

| 列名 | 类型 | 约束 | 示例 |
|------|------|------|------|
| `shot_id` | 文本 | 在整份清单中必须唯一 | `ep01_s01_001` |
| `prompt_en` | 文本 | 建议至少 10 个字符 | `林悦走进咖啡馆...` |
| `duration` | 数字 | 4–15（秒） | `5` |
| `aspect_ratio` | 文本 | 见下方支持列表 | `16:9` |

**支持的画面比例：** `16:9` · `9:16` · `1:1` · `4:3` · `3:4` · `21:9`

短剧竖屏（抖音 / TikTok / Reels）用 `9:16`，标准剧集或宽屏用 `16:9`。

### 可选列——按需使用

| 列名 | 用途 |
|------|------|
| `episode`、`scene_id` | 用于分组管理，不会发送给 API |
| `continuity_group` | 跨镜头保持角色 / 服装 / 场景一致性 |
| `character_ref` | 角色参考图，文件名或 `https://` 链接 |
| `outfit_ref` | 服装参考图 |
| `scene_ref` | 背景 / 场地参考图 |
| `negative_prompt` | 反向提示词，如 `水印, 变形面部, 模糊` |
| `camera` | 镜头运动描述，如 `中景跟拍` |
| `motion` | 主体动作描述，如 `缓慢走进后停顿` |
| `mood` | 情绪基调，如 `宁静亲密的早晨` |
| `prompt_cn` | 中文提示词（`prompt_en` 为空时作为备用） |

### 一行生产级写法示范

```csv
episode,scene_id,shot_id,continuity_group,character_id,character_ref,outfit_ref,scene_ref,prompt_en,camera,motion,mood,duration,aspect_ratio,negative_prompt
ep01,cafe_scene,ep01_s01_001,ep01_s01_lin_cafe,char_lin,char_lin.jpg,outfit_white_coat.jpg,cafe_morning.jpg,"Lin Yue, mole below left eye, pearl stud earrings — walks into the corner cafe and stops at the window table, turns toward camera with a soft expression",medium tracking shot,slow walk-in then pause and gaze,calm intimate morning,5,16:9,"distorted face, extra fingers, watermark, low quality"
```

提示词以角色固定特征开头（`左眼下方痣，珍珠耳钉`），接着是动作、镜头和氛围。这个结构是保持角色一致性效果最好的写法。

## 并发数：同时跑多少条镜头

**并行镜头数**（默认：5）就是同时在处理的任务数量。把每个槽位理解成一个"工人"：拿到一条镜头、提交、等结果、下载，然后再拿下一条。

```
工人:  [镜头1] [镜头2] [镜头3] [镜头4] [镜头5]  ← 5 个并发
       [镜头6] [镜头7] [镜头8] ...               ← 前面有空位就继续
```

:::warning 从小开始
**永远从 5 开始。** 并发太高会触发你密钥的每分钟请求限制（RPM），大量 429 错误反而会让整体速度更慢。
:::

**什么时候可以调高：**
- 快速测试没有出现 429 错误
- 控制台里你的密钥 RPM 设置在 60 以上（控制台 → 密钥 → 编辑）
- 100+ 镜头的大批次，时间比较重要

**推荐调高节奏：** 5 → 8 → 12 → 15，每步观察是否有 429 错误。

## 分步操作：第一次跑 100 条镜头

### 第一步：先验证

在运行之前，点击 **🔍 验证 CSV**。

Batch Studio 会检查：
- 所有必填列都存在
- `duration` 在 4–15 之间
- `aspect_ratio` 是支持的值
- 所有 `shot_id` 都唯一
- 引用的参考图要么已经上传到侧边栏，要么是 `https://` 链接

**错误（红色）** — 该行不会被提交  
**警告（黄色）** — 会提交，但可能不是你预期的效果

在 Excel 里修好错误再重新上传，警告则自行判断是否需要处理。

### 第二步：快速测试，必做

点击 **⚡ 快速测试（3 条镜头）**，再跑完整批次。

只花 3 条镜头的费用，2–5 分钟就能确认：
- 密钥没问题
- 参考图能正常识别
- 前几条镜头的画面质量符合预期
- 再花 100 条镜头的钱是值得的

:::tip 30 分钟换几小时
提前 30 分钟做预检，能节省重新拍的几小时。别跳过快速测试。
:::

### 第三步：完整批次

点击 **▶ 开始完整批量**。保持浏览器标签页开着。

**在并发数 = 5 时的预估时间：**

| 镜头数 | 预估时间 |
|--------|---------|
| 15 条  | 5–15 分钟 |
| 50 条  | 15–30 分钟 |
| 100 条 | 25–50 分钟 |

高峰时段服务端队列比较深，可能比预估多 20–30%。

## 看懂实时进度表

批次运行时，结果表格会逐行更新：

| 列名 | 含义 |
|------|------|
| **镜头 ID** | 对应 CSV 里的哪一行 |
| **状态** | 当前状态（见下方） |
| **任务 ID** | 服务端返回的 Job ID，提交支持工单时需要 |
| **积分** | API 预估的本条镜头费用 |
| **文件** | 下载完成后，本地 MP4 的文件名 |
| **错误** | 失败时的错误码 |
| **详情** | 通俗易懂的错误说明 |
| **继承字段** | 由连贯组自动填入的字段（蓝色高亮显示） |

**状态流转：**

```
⏳ 等待中 → 📤 已提交 → 🎬 生成中 → 💾 已下载
                                  ↘ ❌ 失败
```

`🎬 生成中` 的镜头正在服务商那边渲染，通常需要 20–90 秒。这是最长的一段等待时间。

## 重试失败的镜头

批次完成后，关注汇总数字：
- ✅ 已下载（成功）
- ❌ 失败（需要处理）

查看表格中的 **错误码** 和 **详情** 列，找出原因：

| 错误码 | 通俗含义 | 修复方式 |
|--------|---------|---------|
| `http_401` | 密钥无效 | 在控制台重新生成密钥 |
| `http_402` | 积分用完 | 控制台 → 计费 → 充值 |
| `http_429` | 请求过于频繁 | 降低并发数后再重试 |
| `content_policy.pre` | 提示词被拦截 | 调整措辞 |
| `timeout` | 超过 15 分钟未完成 | 服务端拥堵，重试通常能成功 |
| `no_video_url` | 接口返回成功但没有视频地址 | 偶发性问题，重试即可 |

**重试操作：** 点击 **🔁 仅重试失败项**，系统会新建一个批次文件夹，只重跑失败的行。成功的镜头不会被重复计费。

可以多次重试，直到所有镜头都成功。

## 输出目录结构

每次运行会创建一个带时间戳的文件夹：

```
output/
└── batch_20260423_143022/       ← 批次开始时间
    ├── ep01_s01_001.mp4         ← 以 shot_id 命名
    ├── ep01_s01_002.mp4
    ├── ep01_s01_003.mp4
    ├── ecom_spin_001.mp4
    └── result_manifest.csv      ← 完整运行记录
```

**`result_manifest.csv`** 是完整的运行档案，包含每条镜头的 `job_id`、`status`、`output_url`、`local_file_path`、`error_code` 和 `error_message`。点击 **⬇ 导出结果** 可以保存或归档。

## 最佳实践

### 提示词越细，效果越稳

对比一下：

```
❌ "一个女人走进咖啡馆"
✅ "林悦，左眼下方有颗小痣，戴珍珠耳钉，米色羊毛大衣——
    走进阳光明媚的角落咖啡馆，停在靠窗的桌子旁，
    侧头望向镜头，表情柔和；中景跟拍；
    温暖的晨光从左侧打来；宁静亲密的氛围"
```

第二段提示词同时给了角色特征、镜头、光线和氛围，模型有更多参考点，输出更加稳定。

### 每行都加统一的反向提示词

建议在每行都加上：

```
水印, 变形面部, 多余手指, 多余肢体, 低质量, 模糊
```

反向提示词越统一，质量越稳定。各行不一致会导致随机出现瑕疵帧。

### 一个场景-角色组合，用一个连贯组

如果林悦在一集里出现在三个不同场景，就建三个连贯组：

```
ep01_s01_lin_cafe
ep01_s03_lin_office
ep02_s02_lin_rain_street
```

**不要**把不同场景放进同一个连贯组——会让模型混淆应该继承的视觉上下文。

### 新参考图先用快速测试验证

用新角色参考图跑 50 条镜头之前，先做一次 3 条镜头的快速测试，确认模型确实在用参考图。参考图质量不好会浪费积分。
