# 从「点生成」到「出视频」——小白能懂的流程

> 对照真实代码分组（`backend/cmd/server/main.go` 里的路由）。不要求你会写代码，只要知道 **数据往哪流**。

---

## 1. 你看到的三种「钥匙」

| 类型 | 长什么样（示意） | 谁在用 | 能干什么 |
|------|------------------|--------|----------|
| **客户业务密钥** | `sk_live_...` | 客户程序、Batch Studio | 调 **`/v1/videos`**、看自己的任务和密钥等（在 `Business` 路由组里）。 |
| **管理员密钥** | `ak_...` | 少数自动化脚本 | 管理密钥、账单等 **管理员接口**（`Admin` 路由组，和客户界面不同）。 |
| **浏览器运维会话** | `ops_...`（存在浏览器里） | admin 网页 | 登录管理台后，请求头里带 **运维会话**，高危操作还要 **邮箱验证码（OTP）**。 |

**不要**把 `sk_` 发给陌生人；**不要**把管理台地址当成客户注册入口。

---

## 2. 客户发一条「生成视频」时，大致发生什么？

下面用白话描述 **主路径**（新版接口 `POST /v1/videos`）：

1. **浏览器或脚本**带着 `Authorization: Bearer sk_live_...` 访问 `https://api.nextapi.top/v1/videos`（HTTPS）。
2. **网关**（`internal/gateway`）先确认：这是有效的业务 Key、没超限速、Body 格式合法。
3. **任务模块**（`internal/job`）会管积分：**先预留**，再排队；必要时还要过 **花费控制、内容审核** 等（失败会返回明确错误码，一般 **不会悄悄吞掉**）。
4. **Worker**（另一个进程 `cmd/worker`）从队列里取出任务，通过 **Provider**（`internal/provider/seedance` 等）去调 **火山 / Ark** 上的模型。
5. 模型跑完后，状态写回数据库；客户可以 **轮询查询** 或通过 **Webhook** 收到通知，最后拿到 **视频地址或失败原因**。

失败时，预留的积分通常会按设计 **退还**（具体以账单与运维文档为准）。

## 3. 为什么还有一个「旧接口」？

代码里 **故意保留** 了老路径，避免已经对接的客户突然断掉：

- **`POST /v1/video/generations`** + **`GET /v1/jobs/:id`**：旧式「创建 + 查任务」。
- **`POST /v1/videos`** 等：新式 **B2B 全流程**（幂等、花费、吞吐等更全）。

**新对接优先看** `backend/api/openapi.yaml` 里的 `/v1/videos`。工具包里 Batch Studio 若仍用旧路径，属于历史兼容，不是两套业务逻辑打架。

---

## 4. Server 和 Worker 为什么要两个程序？

可以把它想成 **前台接单** 和 **后厨做菜**：

| 程序 | 比喻 | 作用 |
|------|------|------|
| **server** | 前台 | 快速响应 HTTP：校验、落库、入队、返回任务号。 |
| **worker** | 后厨 | 慢慢跑：调上游、重试、更新进度。 |

这样网页不会因为「某一单生成要 5 分钟」而卡死。

---

## 5. 数据库和 Redis 是干什么的？

- **PostgreSQL**：存用户、组织、密钥、任务状态、账单流水等 **要永久记住** 的东西。  
- **Redis**：队列、部分限流与计数等 **要快、可丢缓存** 的东西（细节可查运维文档）。

本地开发常用 Docker 把这两个一起拉起来（见根目录 `README` 的 `make dev`）。
